| 配置 vite 插件 | ✅ | VitePress 基于 Vite，插件生态兼容 |

### ⚠️ 需要注意的点
1. **搜索配置**：如果用 VitePress 内置 `local-search`，需要额外安装 `vitepress-plugin-search`。建议改用 VitePress 1.x 自带的 `themeConfig.search.provider: 'local'`（无需额外插件），从 1.0 版本开始已内置支持。
2. **base 路径**：如果部署在 `username.github.io` 根域名下，base 设为 `/`；如果在 `username.github.io/repo/` 子路径下，base 要设为 `/repo/`。这个容易忘记，建议一开始就确定部署方式。
3. **Cloudflare**：纯可选，你不配置 Cloudflare 站点一样能跑，只是少了一层 CDN。

## 第三阶段：文章内容管理 — ✅ 全部安全

| 条目 | 评估 | 说明 |
|------|------|------|
| 文章目录组织 | ✅ | 纯文件系统操作 |
| Frontmatter 格式规范 | ✅ | YAML frontmatter 是 Markdown 通用标准 |
| 标签系统实现 | ⚠️ | **这是全计划中少数需要自定义逻辑的地方**。VitePress 没有内置标签系统，需要通过 `useData()` 手动遍历所有页面生成标签聚合页。纯前端操作，不依赖后端。 |
| 分页功能 | ⚠️ | **类似地，分页需要手动实现或找插件**。静态站点分页本质上是限制每页显示的文章数量 + 切片，完全用前端 JS 数组操作可完成。 |

### ⚠️ 需要注意的点
1. **标签系统**：VitePress 本身是文档站点框架，不是专门的博客框架。标签分类需要写一些自定义 Vue 代码来收集 frontmatter 中的 tags 字段并生成标签页。这需要一定的 Vue 能力，但技术上是标准的前端逻辑。**不存在环境依赖问题**。
2. **分页**：静态站点分页有两种方式：
   - 方式 A（推荐）：客户端分页 — 获取所有文章数据，在浏览器端用 JS 分片渲染。简单、稳定、加文章不需要重新构建分页逻辑。
   - 方式 B：构建时分页 — 需要写 VitePress 插件在构建阶段生成多个页面。更复杂，但 SEO 更好。
   - 建议先用方式 A，后期有 SEO 需求再升级到方式 B。

### 扩展性
- 增加新文章 → 新建 `.md` 文件 + 写 frontmatter
- 增加新标签 → 直接在文章的 `tags` 列表里添加，系统自动聚合
- 分页每页数量 → 改一个数字变量

| 文章卡片 | ✅ | 纯 CSS 组件 |

| 标题自动锚点 | ✅ | VitePress 内置支持 |
| 代码块样式 | ✅ | 纯 CSS |
| 图片灯箱 | ⚠️ | 分析到原站使用 **Fancybox UI**。这是一个成熟的第三方库，CDN 引入即可。兼容性好。但如果不想引入额外库，也可以用纯 CSS 放大预览替代。 |
| 页脚 | ✅ | 纯 HTML + CSS |
| 回到顶部 | ✅ | 纯 JS + CSS |
| 分页组件 | ✅ | 纯 Vue 组件 |
| 搜索功能 | ✅ | VitePress 内置 local search 或自定义弹窗 |

### ⚠️ 需要注意的点
1. **Canvas 波浪**：需要在 Vue 的 `onMounted` 中获取 canvas 上下文并绘制。纯客户端操作，SSG 构建时不会执行（需要做 `if (typeof window !== 'undefined')` 判断）。这是标准做法，无兼容问题。
2. **Fancybox**：如果选择引入，注意它需要在 DOM 渲染完成后初始化。建议使用 Vue 的 `nextTick()` 或动态 import。另一个方案是使用 `medium-zoom`（更轻量，只有 zoom 功能无相册），或直接用 CSS 实现简单放大。
3. **Vue 组件在 VitePress 中**：所有组件在构建时会被 SSR（服务端渲染），因此要确保组件代码在 SSR 环境下不会报错。标准做法：访问 `window`/`document` 的地方包在 `onMounted` 或 `if` 判断中。

### 扩展性
- 增加新的 UI 组件 → 新建 `.vue` 文件 → 在 Layout 中引入
- 增加新的页面区域 → 修改 Layout.vue 模板

## 第六阶段：评论系统 — ✅ 基本安全

| 条目 | 评估 | 说明 |
|------|------|------|
| GitHub 注册 OAuth App | ✅ | 标准 GitHub 操作，一次配置永久使用 |
| 引入 Gitalk CSS/JS | ✅ | CDN unpkg.com 加载，稳定可靠 |
| 创建 Gitalk 实例 | ✅ | 纯前端 JS |
| 挂载到文章页底部 | ✅ | 标准 DOM 操作 |
| Gitalk 主题适配 | ⚠️ | Gitalk 的深色模式适配需要覆盖其 CSS 变量，可能需要加 `!important` 或更高优先级选择器，但完全可以实现 |

### ⚠️ 需要注意的点
1. **Gitalk 限制**：Gitalk 依赖 GitHub Issues 作为存储后端。这意味着：
   - 评论者必须要有 GitHub 账号
   - 在 GitHub Issues 页面能看到所有评论（公开数据）
   - 如果仓库设为 private，Gitalk 评论不可用
   - 存在 GitHub API 请求频率限制（未认证 60次/小时，认证后 5000次/小时）
2. **替代方案**：如果觉得 Gitalk 不够灵活，可以考虑：
   - **Waline**：自部署评论系统，支持匿名评论，Node.js 后端
   - **Twikoo**：类似 Waline，需要云函数部署
   - **Giscus**：基于 GitHub Discussions，比 Gitalk 更新、体验更好
   - **Utterances**：类似 Gitalk，更轻量
   - **直接注释掉**：评论功能不是博客必需的，可以后期再加

### 扩展性
- 更换评论系统 → 只需替换评论组件，不影响其他代码
- 增加多种评论后端 → 可以在评论组件中同时加载多个

## 第七阶段：附加功能 — ⚠️ 部分需注意

| 条目 | 评估 | 说明 |
|------|------|------|
| 背景音乐（audio 标签） | ✅ | 标准 HTML5 audio API |
| 播放/暂停按钮 | ✅ | 纯前端 |
| Splash 加载动画 | ✅ | 纯 CSS + SVG |
| Canvas 烟花特效 | ⚠️ | 纯 Canvas API，但粒子系统代码量较大，可能需要调试 |
| Spine 角色 | ❌ | **这是整个计划中唯一需要特殊资源的功能** |

### ❌ Spine 角色 — 不建议纳入第一阶段
Spine 角色功能的问题：
1. **需要 Spine 动画源文件**：`.skel`/`.json` + `.atlas` + 贴图资源。这些资源通常需要购买或请画师制作，不是免费/开箱即用的。
2. **Spine 许可证**：Spine Runtime 虽然是开源免费的，但 Spine 编辑器是商业软件。
3. **替代方案**：如果只是想要页面上的看板娘效果，可以用：
   - **Live2D Cubism**（类似需求，也需要资源文件）
   - **纯 CSS/JS 实现简单角色动画**（无需额外资源）
   - **直接省略该功能**（原站的 Spine 角色虽然有趣，但不是博客核心功能）

建议：Spine 角色标注为"可选中的可选"，放在最后做。

### ⚠️ 需要注意的点
1. **背景音乐自动播放**：绝大多数现代浏览器禁止自动播放音频，需要用户先点击页面任意位置才能播放。这是浏览器策略，无法绕过。实现时需要用 `click` 事件触发 `audio.play()`。
2. **Splash 动画**：VitePress 是 SSG（静态站点生成），页面加载通常很快，Splash 可能在用户看到之前就消失了。建议：
   - 给 Splash 设置一个最小展示时间（例如 800ms）
   - 或：如果构建后页面加载确实很快，可以降低 Splash 优先级
3. **烟花特效**：Canvas 粒子系统是标准前端技术，所有现代浏览器支持。主要问题是性能调优（粒子数量过多时低端设备会卡顿）。建议：
   - 限制粒子数量（50-100 个）
   - 提供开关给用户关闭
   - 移动端自动降低粒子数量或禁用

### 扩展性
- 附加功能都设计为独立组件，可以随时添加/移除而不影响核心功能
- 每个功能都有独立的开关控制

## 对"方便增加条目"的保障分析

该计划在设计上天然支持扩展：

| 扩展场景 | 方式 | 难度 |
|----------|------|------|
| 增加文章 | 新建 `.md` 文件 | ⭐ 极简 |
| 增加导航菜单 | 修改 config 数组 | ⭐ 极简 |
| 增加标签 | 在文章 frontmatter 中加 | ⭐ 极简 |
| 增加 UI 组件 | 新建 `.vue` 文件 + 引入 | ⭐⭐ 简单 |
| 增加 CSS 变量 | 在 `:root` 中加一行 | ⭐ 极简 |
| 增加分类目录 | 新建文件夹 | ⭐ 极简 |
| 增加插件 | npm install + config 注册 | ⭐⭐ 简单 |
| 增加页面 | 新建 `.md` 文件 | ⭐ 极简 |
| 更换评论系统 | 替换 Gitalk 组件 | ⭐⭐ 简单 |
| 增加统计功能 | 在 Layout 中加一段 script | ⭐ 极简 |
| 增加新功能组件 | 在 Layout.vue 中添加 | ⭐⭐ 简单 |

**结论**：该计划的架构设计使得扩展非常方便。所有功能都通过 Vue 组件模块化、CSS 变量主题化、Markdown 文件内容化来组织，新增元素不会影响现有功能。

## 最终结论

**该搭建计划是可行的、稳定的、且易于扩展的。**

- 全计划中约 **85%** 的条目是零风险的标准操作
- 约 **12%** 的条目需要留意但都有成熟的解决方案

计划中没有任何依赖"实验性技术"、"小众环境"或"不稳定 API"的条目。所有技术栈（VitePress、Vue 3、Vite、CSS 变量、Canvas、GitHub Pages、GitHub Actions、Gitalk）都是经过生产环境验证的成熟方案。
